## Goal

Lay the typed-Go query layer on top of the four migrations from 0008–0011. Concretely:

1. Author a focused set of sqlc query files (one per table) covering the access patterns downstream tasks need: identity CRUD, session lifecycle, raw-event upserts (re-poll idempotency), daily-rollup upserts + range scans, sync-run + cursor bookkeeping.
2. Run `sqlc generate` to produce typed Go code at `internal/storage/sqlite/sqlitedb/`.
3. Make the generated `*sqlitedb.Queries` reachable from the rest of the app: add `Querier()` to the sqlite `Store` and register an fx provider so injecting a `*sqlitedb.Queries` "just works".
4. Embed migration SQL into a small `migrations` Go package so tests can apply migrations against an in-memory DB without filesystem path tricks. Update `cmd/migrate` to use the same embedded FS so the binary stays self-contained.
5. Write a single `repo_test.go` that boots an in-memory DB, applies migrations, and exercises representative round-trips per domain (identity, raw event upsert, daily rollup, sync state) — proving sqlc-generated bindings line up with the schema.

This task does **not** try to be exhaustive. Each downstream feature task adds its own queries on demand.

Spec reference: `docs/superpowers/specs/2026-05-08-tempo-design.md` lines 80–122 (the storage section).

## Design decisions

- **Named params**: queries use `@name` syntax (native SQLite parameter form). sqlc generates `*Params` structs with named Go fields, which is dramatically more readable than 14 positional `?` placeholders for `pull_requests`.
- **Upsert shape**: raw events and daily rollups use `INSERT ... ON CONFLICT (<pk>) DO UPDATE SET ... = excluded.<col>` for re-poll / re-aggregate idempotency. We rely on the composite PKs from 0009/0010 as the conflict targets.
- **What `Querier()` returns**: a fresh `*sqlitedb.Queries` bound to the store's `*sql.DB`. Callers that need transactions do `tx, _ := store.DB().BeginTx(ctx, nil); q := sqlitedb.New(tx)` themselves — the generated `*Queries` type has a `WithTx(*sql.Tx)` method too, so we don't need a separate "txn helper" in this task.
- **Why `migrations` becomes a Go package now**: tests need the SQL applied against in-memory SQLite, and the only clean way is `embed.FS`. Embedding requires the `.sql` files to be in the same directory as the Go source that uses `//go:embed`. Co-locating is the obvious choice and lets the binary stay self-contained.
- **No `WithTx` plumbing in this task**: the txn-aware repo wrapper that downstream auth/ingest will want is left for those tasks to land. Generated code already supports it; we just don't build sugar on top yet.
- **Test isolation**: each test gets its own in-memory DB (`file::memory:?cache=shared&mode=memory&_pragma=...`) plus a unique connection-pool key, so parallel tests don't share state. `cache=shared` is required so multiple connections in one pool see the same in-memory DB.

## Steps

### 1. Make `migrations/` a Go package

Create `migrations/migrations.go`:

```go
// Package migrations embeds the goose SQL migration files so they can be
// applied without depending on the working directory.
package migrations

import (
	"context"
	"database/sql"
	"embed"
	"fmt"

	"github.com/pressly/goose/v3"
)

//go:embed *.sql
var FS embed.FS

// Apply runs all up migrations against db using the embedded FS.
func Apply(ctx context.Context, db *sql.DB) error {
	if err := goose.SetDialect("sqlite3"); err != nil {
		return fmt.Errorf("migrations.Apply: dialect: %w", err)
	}
	goose.SetBaseFS(FS)
	defer goose.SetBaseFS(nil)
	if err := goose.UpContext(ctx, db, "."); err != nil {
		return fmt.Errorf("migrations.Apply: up: %w", err)
	}
	return nil
}
```

Update `cmd/migrate/main.go` to call `goose.SetBaseFS(migrations.FS)` and pass `"."` as the directory. Keep the up/down/status/version dispatch the same.

Commit: `feat(migrations): embed SQL files into Go package`.

### 4. Wire `Querier()` and fx provider

In `internal/storage/sqlite/sqlite.go`:

```go
func (s *Store) Querier() *sqlitedb.Queries {
	return sqlitedb.New(s.db)
}

// NewQueries is an fx provider that builds a Querier from a Store.
func NewQueries(s storage.Storage) *sqlitedb.Queries {
	return sqlitedb.New(s.DB())
}
```

We can register `NewQueries` in the server fx graph in a follow-up; this task just lands the function.

## Notes

- `sqlc generate` is idempotent — running it twice produces the same files. Verify-mode lets us catch the case where the committed output diverges from what the queries would produce.
- The existing `verifyPragmas` in `sqlite.go` checks `foreign_keys=1`. Per the project rule we don't *use* FKs in the schema, so the pragma being on is harmless (nothing to enforce). Don't change it here.
- For test DB setup, use `t.TempDir() + "/repo_test.db"` and a normal file path in `cfg.Database.DSN`. Avoid `:memory:` with the shared cache trick — it has subtle pool-state issues that bite under `t.Parallel()`.
- This task only adds queries needed by 0013–0044+. If a feature task needs a query that isn't here, that task adds it and runs `make sqlc-generate` again.
