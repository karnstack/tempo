# 0036 — Review latency + load rollup

  `ListReviewsWithPRMetaForRepo :many`. Returns `(pr_number,
  reviewer_gh_user_id, submitted_at, pr_created_at)` for every
  non-self non-ghost review in the repo, no time filter. The
  aggregator does both the "first review per PR" reduction and the
  `[from, to)` bucketing in Go.

  `DeleteDailyReviewLoadByDateRepo :exec` for the per-repo
  DELETE+INSERT idempotency pattern.
- `internal/storage/sqlite/sqlitedb/*` — regenerated by sqlc.
- `internal/rollup/reviewstats/aggregator.go` — single `Aggregator`
  owns both `daily_review_latency` (one row per (date, repo)) and
  `daily_review_load` (one row per (date, repo, reviewer)). For each
  non-archived repo: bulk-fetch all eligible reviews, derive both
  metrics in Go, then atomically swap the day's slice inside one
  tx (latency UPSERT + load DELETE/INSERTs).
- `internal/rollup/reviewstats/module.go` — fx module providing
  `*Aggregator` into the `"rollup.aggregators"` value group.
- `internal/rollup/reviewstats/aggregator_test.go` — 10 tests
  (see commit message for the full list).
- `cmd/tempo/main.go` — `reviewstats.Module` registered alongside
  the existing rollup modules.

## Notes / followups

- **modernc.org/sqlite + aggregates + TIMESTAMP = pain.** Original
  plan was a SQL-side `MIN(submitted_at) GROUP BY pr.number HAVING
  ... BETWEEN ?from AND ?to`. Two layers of trouble:
  1. sqlc-sqlite types `MIN()` on a TIMESTAMP column as `interface{}`
     because the aggregate hides the column's type.
  2. Even bypassing sqlc with a raw Go const, the modernc.org/sqlite
     driver hands back aggregate-TIMESTAMP values in Go's
     `time.Time.String()` format (`"YYYY-MM-DD HH:MM:SS +ZZZZ TZN"`),
     which Scan can't unmarshal into `time.Time`. Trying
     `unixepoch(MIN(ts))` returns NULL — that format isn't an ISO
     one SQLite's date functions recognise.

  Resolution: one bulk query, all aggregation in Go. Future
  aggregators that need MIN/MAX on TIMESTAMP should default to this
  pattern; comment in the aggregator package docs documents the
  surprise.
- **`Em-dash` (`—`) breaks sqlc-sqlite's lexer.** Same lesson as
  0035; needed twice in this task too. SQL comments must be ASCII.
- **Self-review and ghost filters live in SQL** (`r.reviewer != 0
  AND r.reviewer != pr.author`), so the aggregator doesn't need
  defensive Go checks beyond the negative-duration guard.
- **Negative durations filtered in Go** for the same clock-skew
  defense as 0035's cycletime.
- **`response_minutes_p50` uses integer-minute truncation** via
  `int64(d / time.Minute)`. A 90-second review counts as 1 minute,
  not 1.5. Sub-minute reviews count as 0, which is the right floor
  for "how fast was the response".
- **Per-repo lifetime fetch.** The new query has no time filter, so
  the per-repo result set grows with review history. v1 sqlite +
  in-process aggregation handles this fine, but if a repo grows past
  ~100k reviews we may want to narrow the SELECT to
  `submitted_at < ?to` (still need the full history before `to` so
  the per-PR MIN is correct) or add a `(pr_repo_id, submitted_at)`
  index.
- **0037 hook is free.** Same shape as 0033 — `Aggregate(ctx, date)`
  is idempotent via DELETE+INSERT (load) and UPSERT (latency).
