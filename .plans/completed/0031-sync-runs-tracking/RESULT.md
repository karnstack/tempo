# 0031 — Sync runs + error tracking + status hook

## What changed

- `internal/storage/sqlite/queries/sync_runs.sql` — four new queries:
  `GetLastSuccessfulSyncRun`, `GetLastFailedSyncRun`,
  `PruneSyncRunsByConnection`, `CountFailedSyncRunsSince`.
- `internal/storage/sqlite/sqlitedb/sync_runs.sql.go` + `querier.go` —
  regenerated by `sqlc generate`.
- `internal/ingest/status.go` — new `Status` struct and `StatusFor`
  helper. This is the per-connection hook `/api/v1/sync/status` (0044)
  will sit on top of.
- `internal/ingest/status_test.go` — covers no-runs / only-success /
  only-failure / mixed scenarios.
- `internal/ingest/scheduler.go` — added `pruneRuns(ctx, connectionID)`
  invoked via `defer` from `syncConnection`. Factored `closingCtx` so
  it and `finishRun` share the cancelled-ctx fallback (5s
  background-derived timeout) so housekeeping still lands when the
  parent ctx is dead.
- `internal/ingest/scheduler_test.go` — added `newSchedulerWithRetention`
  helper + `TestTick_PrunesSyncRunsToRetention`. Existing tests routed
  through the new helper with retention=200.
- `internal/config/config.go` — added `Poll.SyncRunRetention int`
  field, parsed from `TEMPO_SYNC_RUN_RETENTION` (default 200),
  validated `>= 1`.
- `internal/config/config_test.go` — default-200 assertion,
  override-50 test, and reject-zero test.

No migration, no main.go wiring change.

## Followups

- 0044 (`/api/v1/sync/status` + `/api/v1/system/health`) is unblocked.
  It will iterate active connections, call `ingest.StatusFor` per
  connection, and roll those into the HTTP response. `/system/health`
  can use `CountFailedSyncRunsSince(now - 1h)` (or similar) for a
  global "is ingest healthy?" boolean.
- 0055 (sync status panel) eventually consumes the same data via the
  generated TS client.
- If the retention default ever feels tight for ops, the env var
  `TEMPO_SYNC_RUN_RETENTION` raises it without code changes.
