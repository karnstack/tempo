## Notes / followups

- **Inclusive date range, half-open queries.** The API treats
  `from`/`to` as inclusive YYYY-MM-DD dates; internally we pass
  `to + 1 day` to the existing `[from_date, to_date)` list
  queries. Documented in the handler.
- **365-day cap.** Per-reviewer daily rows in `daily_review_load`
  multiply by active-reviewer count, so a multi-year query for a
  busy repo could pull thousands of rows. Cap matches the
  highest-bound dashboard scenario (annual review).
- **Cross-tenant returns 404, not 403.** Matches the connections
  handler's stance — opaque ids and existence shouldn't leak across
  tenants.
- **No aggregation in the response.** Per-day rows go out as
  stored; the frontend handles summing for headline numbers.
  Keeps the handler simple and gives the dashboard the raw series
  it needs for charts.
- **`tenantIDFromSession` duplicated.** Third copy now (tokens,
  connections, repos). The first 0042/0043 handler that needs it
  should justify a `internal/api/web/session.go` extraction.
- **Archived repos still serve their history.** Archiving stops
  ingest but the rollup data remains; the dashboard can show "what
  did this repo do in its prime?".
