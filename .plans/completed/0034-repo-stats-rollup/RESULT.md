# 0034 — Repo stats rollup

## Files changed

- `internal/storage/sqlite/queries/daily_repo_stats.sql` — added
  `AggregateRepoStatsForDay :exec`. Tables are aliased (`pr`, `d`) so
  sqlc-sqlite can disambiguate `created_at` across `pull_requests`
  and `deployments` under one INSERT.
- `internal/storage/sqlite/sqlitedb/daily_repo_stats.sql.go`,
  `internal/storage/sqlite/sqlitedb/querier.go` — regenerated by sqlc.
- `internal/rollup/repostats/aggregator.go` — `Aggregator` type with
  `Name()` / `Aggregate()`. No per-repo tx: the UPSERT is a single
  statement so SQLite handles atomicity on its own.
- `internal/rollup/repostats/module.go` — fx module providing
  `*Aggregator` into the `"rollup.aggregators"` value group.
- `internal/rollup/repostats/aggregator_test.go` — 7 tests covering
  empty DB, all-sources happy path, no-double-counting of merged PRs,
  multi-repo partitioning, archived-repo skip, window boundaries,
  idempotent rerun + stale counts cleared, and the disjoint-columns
  contract with 0035 (lead-time columns preserved across a re-run).
- `cmd/tempo/main.go` — `repostats.Module` registered alongside the
  ingest modules and `engineerstats.Module`.

## Notes / followups

- **sqlc-sqlite needs qualified columns**, not raw-Go SQL. Unlike
  0033's CTE-chained `WITH ... INSERT ... SELECT` (which sqlc-sqlite
  flat-out can't parse), 0034's `INSERT VALUES (...COUNT subqueries...)`
  *did* parse — sqlc just needed table aliases (`pr.created_at` vs
  `d.created_at`) so it could resolve the column reference. Worth
  trying explicit aliases before falling back to raw SQL in future
  aggregators.
- **Disjoint-columns contract enshrined.** The aggregator's
  `ON CONFLICT DO UPDATE` deliberately omits
  `lead_time_seconds_p50/p90`, and there's a dedicated test
  (`TestAggregate_LeadTimeColumnsPreserved`) that asserts the
  preservation. 0035 should mirror this — UPSERT with its own column
  list, never a leading DELETE on this row.
- **`prs_closed` excludes merges.** The subquery filters
  `merged_at IS NULL AND closed_at IS NOT NULL`, so a PR merged today
  counts as `prs_merged` only — never both. The test
  `TestAggregate_MergedPRNotDoubleCounted` locks this in.
- **No status filter on `deploys`.** Deployment rows are counted
  regardless of `status` (which is empty in v1 — see
  `internal/ingest/deployments/doc.go:42-44`). If we ever start
  populating status from `/deployments/{id}/statuses`, the aggregator
  can grow a `status = 'success'` filter.
- **0037 hook is free.** Same shape as 0033 — re-running a date is
  `Aggregate(ctx, date)`, idempotent by UPSERT.
